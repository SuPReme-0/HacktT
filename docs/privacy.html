Privacy Policy & Data Operation

Last Updated: March 22, 2026 | Version 1.0.0

🛡️ Core Principle: HackT Runtime is a zero-knowledge, local-first security agent. All AI reasoning, threat detection, and vault storage occur strictly on your local hardware. We do not transmit your conversations, code snippets, screen captures, or threat intelligence to our servers.

1. Data We Collect

1.1 Authentication Data (Cloud-Stored)

To secure your local Knowledge Vault and verify your identity across devices, we use Google Single Sign-On (SSO) via Supabase Auth. We collect and store ONLY the following data in our secured cloud database:

  • Email Address – For account recovery and session management
  • Public Google Profile Name – Displayed in the operator dashboard
  • Public Google Avatar URL – Displayed in the operator dashboard
  • Session Tokens – Encrypted JWT tokens for authentication (expires after 30 days)

1.2 Local-Only Data (Never Transmitted)

The following data is stored exclusively on your device in encrypted local databases (LanceDB, KùzuDB, SQLite):

  • Chat History – All conversations with the AI agent
  • Code Snippets – Any code analyzed or generated by HackT
  • Screen Captures – Florence-2 OCR outputs and threat detections
  • Voice Recordings – Faster-Whisper STT transcriptions (processed locally, not stored)
  • Threat Intelligence – Detected vulnerabilities, phishing attempts, and security alerts
  • Project Context – File paths, IDE context, and browser history analyzed by HackT
⚠️ Important: If you enable Cloud Sync in Settings, encrypted chat history backups will be uploaded to your Supabase bucket. This is optional and disabled by default. All data is encrypted with AES-256 before transmission.

2. System Permissions & Local Processing

To function as an active security monitor, the HackT desktop application requests specific Windows permissions during installation. This data never leaves your machine:

2.1 Screen Capture (OCR)

Used by our local Florence-2 Vision model to read your IDE or browser for real-time threat detection. Screen frames are processed in RAM and discarded immediately after analysis. No screenshots are stored unless explicitly requested by the operator.

2.2 Microphone Access

Used strictly for local Faster-Whisper speech-to-text (STT) voice commands. Audio is processed in real-time on CPU. No audio recordings are stored or transmitted.

2.3 Network Proxy (Ports 8081, 8082)

Used to establish WebSocket connections with VS Code / JetBrains extensions (Port 8081) and Chrome browser extension (Port 8082) for real-time code and traffic analysis. All communication occurs over localhost (127.0.0.1).

2.4 Administrator Privileges

Required for:

  • Installing the Tauri runtime and Python sidecar
  • Registering the `hackt://` deep link protocol with Windows
  • Enabling OS-level keyboard injection for code fixes (BlockInput API)

3. How We Use Your Data

  • Authentication: Email and profile data are used solely to verify your identity and manage your Supabase session.
  • Local AI Inference: All conversations and code analysis are processed locally by Qwen 3.5 4B and Florence-2. No data is sent to external AI APIs.
  • Threat Detection: Screen captures and browser traffic are analyzed locally for phishing, XSS, SQL injection, and other vulnerabilities.
  • Vault Learning: Your local Knowledge Vault (LanceDB + KùzuDB) learns from your projects to provide context-aware responses. This data never leaves your device.

4. Data Retention & Deletion

4.1 Local Data

All local data is stored in %AppData%/HackT/ on Windows. You can delete this folder at any time to remove all chat history, vault data, and settings. The application will regenerate default configurations on next launch.

4.2 Cloud Data

Authentication data (email, profile name, avatar) is retained until you request account deletion. Contact us at priyanshuroy0912@gmail.com to request permanent deletion of your Supabase account.

4.3 Optional Cloud Sync

If enabled, encrypted chat history backups are stored in your Supabase bucket. You can disable Cloud Sync at any time in Settings. Existing backups can be deleted via the Supabase dashboard or by contacting support.

5. Third-Party Services

HackT Runtime uses the following third-party services:

  • Supabase Auth – For Google SSO authentication (data: email, profile name, avatar)
  • Google OAuth – For identity verification (data: public profile info only)
  • GitHub Releases – For software distribution (data: download analytics, IP address)

No other third-party services have access to your data. We do not use analytics, tracking cookies, or advertising networks.

6. Security Measures

  • Encryption at Rest: Local vault data is encrypted with AES-256
  • Encryption in Transit: All cloud communication uses TLS 1.3
  • Session Management: JWT tokens expire after 30 days of inactivity
  • Code Signing: All executables are signed with a valid Windows certificate
  • Open Source: All source code is available on GitHub for public audit

7. Children's Privacy

HackT Runtime is not intended for users under 18 years of age. We do not knowingly collect data from children. If you believe a child has provided us with personal data, please contact us immediately.

8. Changes to This Policy

We may update this Privacy Policy from time to time. Significant changes will be announced via the HackT dashboard and on this page. Continued use of the software after changes constitutes acceptance of the updated policy.
